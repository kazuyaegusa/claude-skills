# 3モジュール型自己改善アーキテクチャを実装する

> 短期記憶・自己批評・自己最適化の3つのモジュールを組み合わせ、AIモデルが自分自身の失敗を記録・分析・改善するシステムを構築する

- 出典: https://x.com/0x__tom/status/2034561616302485658
- 投稿者: Tom | ドバイで生成AIやってる人
- カテゴリ: other

## なぜ使うのか

単一のモデルでは過去の失敗パターンを体系的に参照できず、同じ間違いを繰り返す。記憶・批評・最適化を分離することで、各ラウンドの知見を蓄積し、次ラウンドで活用できる

## いつ使うのか

モデルの性能を継続的に改善したいが、人間がラベリング・評価・調整を行うコストが高すぎる場合。特にRL・コード生成・エージェントタスクで有効

### 具体的な適用場面

- 社内AIエージェントの性能を人手を介さず継続的に改善したい場合
- RL研究・モデル最適化タスクの自動化により研究開発コストを削減したい場合
- バグ修正・コード最適化を自律的に実行させ、エンジニアリングリソースを戦略的タスクに集中させたい場合
- 温度パラメータやプロンプト戦略の探索を自動化し、最適設定を発見したい場合

## やり方

1. 短期記憶モジュール: 各ラウンドの実行結果（成功/失敗、エラーメッセージ、生成コード）をJSON形式で保存する
2. 自己批評モジュール: 保存された記憶に対して「何がダメだったか」を分析するプロンプトを実行し、改善提案を生成する
3. 自己最適化モジュール: 過去全ラウンドの記憶と批評を参照し、次の改善案（コード修正・パラメータ変更）を生成する
4. 評価関数で新旧バージョンを比較し、性能向上が確認できた場合のみ採用する
5. 100ラウンド以上繰り返し、収束するまで継続する

### 入力

- 初期モデル（ベースライン）
- 評価セット（性能測定用テストケース）
- 失敗ログを記録するストレージ

### 出力

- 改善されたモデル（または最適化されたコード）
- 各ラウンドの性能変化ログ
- 採用された改善パターンのリスト

## 使うツール・ライブラリ

- MiniMax M2.7 API
- autoresearch（Karpathy OSS、参考実装）
- OUROBOROS（個人実験、アーキテクチャ参考）

## 前提知識

- LLMのAPI呼び出しとプロンプトエンジニアリングの基礎知識
- 強化学習（RL）の基本概念（報酬・エピソード・評価）
- コード生成モデルの評価指標（pass@k、SWE-bench等）への理解
- JSON形式でのログ記録・データ管理の実装経験

## 根拠

> M2.7は「失敗を分析 → コード修正 → 評価 → 比較 → 採用or破棄」のループを100ラウンド以上、完全自律で回した

> 短期記憶、自己批評、自己最適化の3モジュール構成

> 毎ラウンド「何がダメだったか」のメモリを生成。次ラウンドは過去全ラウンドの記憶を参照して改善

> 温度パラメータの最適組み合わせも自分で探索

> バグ修正後に同パターンを他ファイルから自動探索する機能まで自力で追加

# グローバルとプロジェクト固有のエージェント使い分け

> `~/.claude/agents/`（グローバル）にチーム共通のエージェントを配置し、`.claude/agents/`（プロジェクト固有）にプロジェクト独自のカスタマイズ版や追加エージェントを配置することで、汎用性と特殊性を両立させる

- 出典: https://github.com/VoltAgent/awesome-claude-code-subagents
- 投稿者: VoltAgent
- カテゴリ: claude-code-workflow

## なぜ使うのか

全プロジェクトで共通のベストプラクティス（セキュリティ監査、コードレビュー基準等）はグローバルで管理し、特定プロジェクトのドメイン知識や技術スタック（例: fintech固有のルール、特定フレームワークの規約）はプロジェクト固有で管理することで、保守性と柔軟性を確保

## いつ使うのか

複数プロジェクトを並行運用している場合、チーム全体で統一されたベストプラクティスを維持しつつプロジェクトごとの特殊要件にも対応したい場合

## やり方

1. チーム全体で共有すべきエージェント（code-reviewer, security-auditor等）を`~/.claude/agents/`に配置
2. プロジェクト固有のエージェント（例: このプロジェクト専用のAPI設計ルールを持つapi-designer-custom）を`.claude/agents/`に配置
3. 名前が衝突した場合、プロジェクト固有のエージェントが優先される
4. チーム全体で共通エージェントを更新する際は、グローバルフォルダのファイルを更新してgit経由で配布

### 入力

- チーム共通のベストプラクティス定義
- プロジェクト固有の要件・制約

### 出力

- グローバルエージェント群（~/.claude/agents/）
- プロジェクト固有エージェント群（.claude/agents/）
- 優先順位に基づく自動選択

## 使うツール・ライブラリ

- Claude Code CLI
- git（チーム間でのエージェント共有用）

## コード例

```
# グローバル（全プロジェクト共通）
~/.claude/agents/
├── code-reviewer.md
├── security-auditor.md
└── devops-engineer.md

# プロジェクト固有（このプロジェクトのみ）
.claude/agents/
├── fintech-engineer-custom.md  # 金融ドメイン固有ルール
└── api-designer-custom.md      # このプロジェクトのAPI設計規約

# 衝突時はプロジェクト固有が優先される
```

## 前提知識

- Claude Code CLI がインストールされていること
- Claude Codeの基本操作（エージェント起動、ツール実行）の理解
- YAMLフロントマターの基本構文知識（エージェントカスタマイズ時）
- gitの基本操作（リポジトリクローン、ファイルコピー）
- 対象プロジェクトの技術スタックと開発フローの把握

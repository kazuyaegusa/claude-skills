# LLMの学習分布に適合した操作を優先する

> カスタムツールを新規定義するのではなく、LLMが既にGitHubコードベースで学習済みの操作（ls、grep、ファイル読み書き）を使う

- 出典: https://x.com/__timakin__/status/2013022798957203554
- 投稿者: Seiji Takahashi - timakin / ベースマキナで社内管理画面作成
- カテゴリ: agent-orchestration

## なぜ使うのか

LLMは膨大なGitHubコードで訓練されており、ファイルシステム操作は「既知の能力」。新しいツールの使い方をプロンプトで説明すると追加のトークンを消費し、実行精度も下がる

## いつ使うのか

AIエージェントに新しい機能を追加する設計段階で、実装方法を決定する時

### 具体的な適用場面

- AIエージェントのコンテキストウィンドウが限界に達し、会話履歴や中間結果の保存方法を検討している時
- MCPカスタムツールを実装しようとしてスキーマ設計・エラーハンドリング・認証実装の複雑さに直面している時
- プロジェクト固有のルールや手順をAIエージェントに伝えたいが、プロンプトに毎回含めるとトークンを圧迫する時
- 複数セッションに渡ってAIエージェントに同じコンテキストを維持させたい時

## やり方

1. AIエージェントに新機能を追加する際、まず `ls`, `grep`, `cat`, `echo >`, `bash` などの基本操作で実現できないか検討 2. 実現できる場合はカスタムツール定義をスキップし、ファイル操作で実装 3. 複雑なロジックが必要な場合のみbashスクリプトに切り出す 4. MCPツールは外部API統合など、ファイル操作では不可能な場合のみ使用

### 入力

- 実装したい機能の要件

### 出力

- シンプルで学習コストのない実装
- トークン消費の削減

## 使うツール・ライブラリ

- ls
- grep
- cat
- bash

## 前提知識

- AIエージェントとLLMの基本的な仕組みの理解
- コンテキストウィンドウとトークン制限の概念
- MCP（Model Context Protocol）の基本知識
- bashとファイルシステムの基本操作（ls, grep, cat等）

## 根拠

> LlamaIndexの記事では、LLMが膨大なコードベースで訓練されている点を強調しています。GitHubに上がっているコードの多くには`grep`や`ls`、ファイルの読み書きといった操作が含まれています。つまり、LLMにとってファイルシステムの操作は、わざわざ教え込む必要のない「既知の能力」になっています。

> Anthropicの事例では、この方式で150,000トークンから2,000トークンへ削減できたと報告されています。98.7%の削減です。

> Claude Codeの`CLAUDE.md`も同じ発想だと思います。プロジェクトのコンテキストをファイルに書いておけば、セッションが変わっても参照できます。

> LlamaIndexはこの流れを「ドキュメントをコンテキストとして与えるだけで、専用ツールと同等の機能が実現できる」と表現しています。

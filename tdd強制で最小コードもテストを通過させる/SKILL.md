# TDD強制で最小コードもテストを通過させる

> 最小単位のサブタスクであっても、実装前にテストを書き（red）、テストが通るまで実装し（green）、リファクタするred/green TDDサイクルを強制する

- 出典: https://x.com/jiroucaigou/status/2035964407201849564
- 投稿者: 努力赚钱的菜狗
- カテゴリ: other

## なぜ使うのか

テストなしで進めると後でデバッグが困難になり、AIが生成したコードの品質保証ができない。YAGNI（今必要なものだけ）とDRY（繰り返しをなくす）を組み合わせることで技術負債を最小化できる

## いつ使うのか

サブエージェントにコーディングタスクを委任するすべての場面。特にAIが書いたコードの品質を保証したい場合

### 具体的な適用場面

- Claude Code / Cursorで新機能を実装しようとするたびにAIが設計なしに書き始めて後悔する状況
- チームでAIコーディングを使っているが、各人がバラバラなプロンプト戦略を取っていて品質が安定しない状況
- AIが生成したコードが動くが後のリファクタ・バグ修正が困難になっているプロジェクト

## やり方

SuperPowersのTDDスキルが自動起動し:
1. サブエージェントがテストを先に書く（red: 失敗するテスト）
2. テストが通る最小限の実装を書く（green）
3. コードをリファクタしてもテストがgreenのまま保つ
4. すべてのテストがpassした場合のみタスク完了とみなす

### 入力

- サブタスクの仕様（設計文書から抽出）

### 出力

- テスト通過済みの実装コード
- テストファイル

## 使うツール・ライブラリ

- obra/superpowers (TDDスキル)

## 前提知識

- Claude Code または Cursor が動作する環境
- Node.js / npx が使える環境（CLIインストールの場合）
- git が使える環境（worktree機能のため）
- AIコーディングエージェントの基本的な使い方を知っていること

## 根拠

> 「SuperPowers制定了一套规则，让AI可以像资深团队一样有一套标准的开发流程。目前已在Github斩获106kstars」

> 「1.先疯狂提问，把你的需求完全理解，你同意后并生成标准设计文档」

> 「2.开新分支+隔离工作区，确保不会搞乱主代码」

> 「3.把任务拆分成最小子任务」

> 「4.交给子Agent执行并强制测试，最小代码也要通过测试」

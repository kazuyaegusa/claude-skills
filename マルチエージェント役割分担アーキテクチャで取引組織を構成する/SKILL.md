# マルチエージェント役割分担アーキテクチャで取引組織を構成する

> アナリスト（4体）、強気/弱気の研究員、トレーダー、リスク管理チームという役割別のAIエージェントを配置し、それぞれが財務諸表・市場センチメント・過去データを分析した上で内部議論させる

- 出典: https://x.com/mubeitech/status/2035250400467497096
- 投稿者: 墓碑科技
- カテゴリ: agent-orchestration

## なぜ使うのか

単一AIは特定の視点に偏りやすいが、役割を分けることで強気・弱気の両論が必ず提示され、トレーダーが総合判断し、リスク管理が最終チェックする人間組織の意思決定プロセスを再現できる。これにより判断の質が向上し、極端なポジション取りを抑制できる

## いつ使うのか

AIによる自動取引システムを構築する際、単一モデルの偏りを防ぎ、多角的な分析と慎重な意思決定プロセスを組み込みたい時

### 具体的な適用場面

- 個人投資家が機関投資家レベルの多角的分析・リスク管理体制を低コストで構築したい時
- 従来のシングルエージェントAI取引システムが過度に楽観的/悲観的な判断をしてしまう問題を、複数視点の議論で緩和したい時
- 財務諸表分析・市場センチメント分析・リスク管理を統合した自動取引システムを構築したい時
- ヘッジファンドや資産運用会社が人件費を削減しつつ24時間稼働の取引体制を整えたい時

## やり方

1. TradingAgentsリポジトリ（GitHub上で3万スター）をクローン 2. config.yamlで各エージェントの役割（analyst, bull_researcher, bear_researcher, trader, risk_manager）とLLMモデル（GPT-5.4/Claude 4.6等）を定義 3. 各エージェントに異なるプロンプト（強気研究員は「ポジティブ材料を重視せよ」、弱気研究員は「リスク要因を強調せよ」等）を設定 4. エージェント間の通信フロー（アナリスト→研究員→トレーダー→リスク管理）をワークフローエンジンで定義 5. システム起動後、各エージェントが並列に情報収集し、内部チャットログで議論、最終的にトレーダーが取引判断し、リスク管理が拒否権を持つ仕組みで稼働

### 入力

- 財務諸表データ（財報）のAPI接続情報
- 市場センチメントデータ（SNS/ニュース）のクローリング設定
- 各エージェントの役割定義（YAML設定）
- 使用するLLMのAPIキー（OpenAI/Anthropic）

### 出力

- 各エージェントの分析レポート（テキスト）
- 内部議論ログ（強気vs弱気の論点整理）
- 最終的な取引指示（買い/売り/保留、数量、ストップロス設定）
- リスク管理チームの承認/拒否判断とその理由

## 使うツール・ライブラリ

- TradingAgents（GitHubリポジトリ名、正確な名称は投稿から不明だが'TradingAgents'と記載）
- GPT-5.4（OpenAI APIモデル名、投稿に明記）
- Claude 4.6（Anthropic APIモデル名、投稿に明記）

## 前提知識

- Python環境（TradingAgentsの実行に必要）
- OpenAI API/Anthropic APIの基本的な使い方
- 金融市場の基礎知識（最大ドローダウン、シャープレシオ等のリスク指標の意味）
- 自動取引システムの法的規制理解（地域により制限がある場合がある）
- APIコスト管理（LLM呼び出しが頻繁に発生するため、コスト試算が必要）
